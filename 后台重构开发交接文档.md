# 后台重构开发交接文档

## 1. 项目目标

把当前项目的后台管理端重构成接近 `/Users/nianbaidediannao/Desktop/pojie` 后台的完整运营后台体验：

- UI 风格参考 `pojie/public/admin`：白底侧栏、卡片式菜单、浅阴影、轻动效、密集但清晰的管理后台布局。
- 功能覆盖参考 `pojie`，但不复制 `pojie` 的接口代码。
- 接口必须继续使用当前项目的接口体系：`admin/src/api/*` 调用当前 Nest 后端 `backend/src/modules/*`。
- 不允许引入 `pojie/routes`、`pojie/service`、`pojie/utils` 等后端接口实现。
- 如果某个 `pojie` 功能当前后端没有接口，应在当前 Nest 后端新增模块、DTO、Service、Controller，并同步前端 API 封装。

## 4. 当前登录 500 的根因

当前用户在 `http://localhost:3001/#/login?redirect=/dashboard` 输入账号密码后看到：

```text
Request failed with status code 500
```

复现命令：

```bash
curl -i -X POST http://localhost:3001/api/admin/login \
  -H 'Content-Type: application/json' \
  --data '{"username":"admin","password":"Admin@123456"}'
```

现象：

- 前端 Vite 服务 `localhost:3001` 已启动。
- 后端 `localhost:3000` 没有监听。
- PostgreSQL `localhost:5432` 没有监听。
- Redis `localhost:6379` 没有监听。
- 启动后端时报错：

```text
PrismaClientInitializationError: Can't reach database server at `localhost:5432`
[ioredis] Error: connect ECONNREFUSED ::1:6379
```

结论：

这个 500 不是前端登录页面重构导致的，而是后端依赖服务未启动。Vite 将 `/api/admin/login` 代理到 `http://localhost:3000/admin/login`，后端不可用时会返回 500 代理错误。

默认后台账号来自 `backend/prisma/seed.ts`：

```text
username: admin
password: Admin@123456
```

## 5. 本地开发启动步骤

### 5.1 启动数据库和 Redis

必须先启动 PostgreSQL 和 Redis，并确保它们与 `backend/.env` 匹配：

```env
DATABASE_URL="postgresql://lingmeng:<password>@localhost:5432/lingmeng_db?schema=public"
REDIS_HOST=localhost
REDIS_PORT=6379
REDIS_PASSWORD=<password>
```

推荐用 Docker/OrbStack：

```bash
cd /Users/nianbaidediannao/Desktop/后端后台文件/backend

# 启动 Docker/OrbStack 后，再运行：
docker run -d --name lingmeng-postgres-dev \
  -e POSTGRES_USER=lingmeng \
  -e POSTGRES_PASSWORD='<与 backend/.env DATABASE_URL 中一致>' \
  -e POSTGRES_DB=lingmeng_db \
  -p 127.0.0.1:5432:5432 \
  postgres:15-alpine

docker run -d --name lingmeng-redis-dev \
  -p 127.0.0.1:6379:6379 \
  redis:7-alpine redis-server --requirepass '<与 backend/.env REDIS_PASSWORD 一致>'
```

如果使用 `backend/docker-compose.yml`，注意它默认不把 PostgreSQL/Redis 端口暴露给宿主机。若后端在宿主机直接运行，必须临时给 `postgres` 和 `redis` 增加 `127.0.0.1` 端口映射，或把后端也放到同一个 compose 网络内运行。

## 6. 后续重构原则

### 6.1 严禁事项

- 不要把 `pojie` 的后端 routes/service/utils 复制进本项目。
- 不要把 `pojie` 打包后的 JS 当源码直接混入当前 Vue 项目。
- 不要绕过 `admin/src/api/*` 直接在页面里写 axios URL。
- 不要破坏当前权限系统：`userStore.hasPermission`、路由 `meta.permission`、后端 AdminGuard/AdminPermissionGuard。
- 不要重置或删除用户已有文件。

### 6.2 推荐做法

每补一个功能模块，按这个顺序做：

1. 从 `admin/POJIE_FEATURE_AUDIT.md` 选一个模块。
2. 查当前前端是否已有页面：`admin/src/views/<module>`。
3. 查当前接口封装是否已有文件：`admin/src/api/<module>.ts`。
4. 查当前后端是否已有 Controller/Service：`backend/src/modules/<module>`。
5. 如果后端已有接口，只补 UI 和 API 封装。
6. 如果后端没有接口，在当前 Nest 后端新增模块。
7. 每个模块完成后运行 typecheck/build，并用 curl 测关键接口。

## 9. API 封装规范

每个模块一个 API 文件：

```ts
// admin/src/api/example.ts
import request from '@/utils/request'
import type { ApiResponse, PageParams, PageResult } from '@/types/api'

export const exampleApi = {
  list(params: PageParams) {
    return request.get<ApiResponse<PageResult<any>>>('/admin/example', { params })
  },
  detail(id: string) {
    return request.get<ApiResponse<any>>(`/admin/example/${id}`)
  },
  create(data: any) {
    return request.post<ApiResponse<any>>('/admin/example', data)
  },
  update(id: string, data: any) {
    return request.put<ApiResponse<any>>(`/admin/example/${id}`, data)
  },
  remove(id: string) {
    return request.delete<ApiResponse<any>>(`/admin/example/${id}`)
  },
}
```

然后在 `admin/src/api/index.ts` 统一 export。

## 10. 后端实现规范

新增模块建议结构：

```text
backend/src/modules/<module>/
  <module>.module.ts
  <module>.controller.ts
  <module>.service.ts
  dto/
    <module>.dto.ts
```

接口路径统一：

```ts
@Controller('admin/<module>')
```

列表接口统一返回：

```ts
{
  list: [],
  total: 0,
  page: 1,
  pageSize: 20
}
```

权限：

- Controller 使用 `AdminGuard`。
- 需要细粒度权限时使用 `AdminPermissionGuard` 和 `@RequirePermission('xxx:view')`。
- 新权限要同步 seed 到 `backend/prisma/seed.ts`。
